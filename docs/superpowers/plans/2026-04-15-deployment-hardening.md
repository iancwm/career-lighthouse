# Design: Deployment Hardening Plan

## Problem Statement
The repo works in the happy-path local Docker setup, but it has too many environment assumptions baked into auth, filesystem ownership, and service endpoints.

The main failures are:
- Admin endpoints depend on `X-Admin-Key`, but the browser admin UI does not send that header.
- Terraform points the API at `http://localhost:6333`, which is not a real Qdrant service in ECS.
- Session storage assumes `/app/data/sessions` is writable, but the runtime ownership of mounted volumes is not guaranteed.
- Some storage code swallows `OSError`, which hides mount and permission problems until the UI looks empty.
- The repo still uses query-param admin auth in places, which leaks into browser history and logs.

The result is brittle deployment behavior. Local Docker can look fine while ECS, Amplify, or any alternate runtime breaks in ways that are hard to diagnose.

## Deployment Target
This hardening pass keeps the current app architecture, file-backed knowledge state, and local Docker workflow.

Target shape by environment:
- Local Docker: `docker-compose.yml` runs `api`, `web`, and `qdrant` together, with bind mounts for repo data and a named volume for vector data.
- ECS / production: `api` runs in Fargate behind an ALB, `web` is deployed on Amplify Hosting, knowledge and session files live on persistent EFS-backed storage, and Qdrant is an external service reached through an explicit `QDRANT_URL`, not `localhost`.

That means the repo must support both:
- container-owned state in dev,
- explicit env-backed state in prod.

## Writable Paths
These are the paths that must be declared, mounted, or initialized in every environment:

| Path | Purpose | Owner | Local Docker | ECS / prod |
|------|---------|-------|--------------|------------|
| `/app/data/sessions` | Session JSON files | API task | named volume or host bind | mounted persistent storage |
| `/app/knowledge/career_profiles` | Career profile YAMLs | API task | repo bind mount | persistent storage |
| `/app/knowledge/employers` | Employer YAMLs | API task | repo bind mount | persistent storage |
| `/app/logs` | query and publish logs | API task | repo bind mount | persistent storage |
| `/app/.cache` | sentence-transformers cache | API task | named volume | ephemeral or mounted cache |
| `/home/appuser/.cache/uv` | uv cache | API task | initialized at container start | ephemeral or mounted cache |
| `/qdrant/storage` | vector DB files | Qdrant service | Qdrant container volume | managed by Qdrant, not the API |

The key point is ownership. If these paths exist but the runtime user cannot write them, the app should fail loudly and immediately.
For Qdrant specifically, the API must not assume it owns vector storage in production. The deployment target either provisions Qdrant as its own service or uses Qdrant Cloud, and the API talks only to the exposed endpoint.

## Constraints
- Local `just up` must keep working.
- `/admin` should remain open when `ADMIN_KEY` is unset in dev.
- Production must not rely on query params for admin access.
- File-backed stores are okay for now, but the writable roots must be explicit and portable.
- The API task should remain single-writer for file-backed sessions until state moves out of files.
- The API should fail loudly when deployment wiring is wrong, not silently return empty data.
- The same codebase must run in dev compose, Amplify + ECS, and a local `uv run` workflow without hidden path assumptions.

## Premises
1. Deployment safety matters more than preserving a convenience path that only works locally.
2. The browser and the API need one shared auth transport that survives reverse proxies and different hosts.
3. All writable paths must be declared by env or mounted volume, not inferred from image ownership.
4. `localhost` inside Terraform is a smell unless the service truly runs in the same container.
5. File-backed sessions can stay for now, but they need a durable root and a single-writer contract.
6. Until state is externalized, the API should be deployed with one task replica and one writer.

## Approaches Considered

### Approach A: Minimal Safe Rollout
Summary: fix the runtime ownership and wiring issues first, keep the current auth shape for now, and leave the deeper auth cleanup for later.

Effort: S
Risk: Low

Pros:
- Smallest diff.
- Gets deployment back on its feet quickly.
- Reduces the number of moving parts during the first pass.

Cons:
- Leaves query-param admin auth in place.
- Leaves some auth coupling between frontend and backend unresolved.
- Makes prod safer, but not clean.

Reuses:
- Current file-backed stores.
- Existing Docker Compose structure.
- Existing healthcheck patterns.

### Approach B: Full Deployment Hardening
Summary: make auth transport explicit, make writable paths explicit, and make Terraform point at real external services instead of localhost assumptions.

Effort: L
Risk: Medium

Pros:
- Closes the exact findings from the audit.
- Makes local Docker, ECS, and Amplify behave from the same config model.
- Surfaces wiring mistakes early instead of hiding them.

Cons:
- Touches more files and more layers.
- Needs careful testing because auth and storage are coupled.
- Requires one clean migration path for admin calls.
- Does not solve multi-writer data races, it only makes the current single-writer assumption explicit.

Reuses:
- Existing `ADMIN_KEY` env pattern.
- Existing `SessionStore`, `CareerProfileStore`, and `EmployerEntityStore` layouts.
- Existing Dockerfile and Terraform scaffolding.

### Approach C: Externalize More State
Summary: move session state and other file-backed state into a managed data service, and treat the repo as a thin app layer.

Effort: XL
Risk: High

Pros:
- Strongest long-term portability.
- Removes a lot of filesystem ownership risk.
- Easier to scale across more than one task or instance.

Cons:
- Overkill for the current repo.
- Too much change for one hardening pass.
- Requires a bigger product decision than the audit calls for.

Reuses:
- The current model and API surface.
- Existing entity serialization.

## Recommended Approach
Choose Approach B, with a two-step rollout.

Phase 1 fixes the hard deployment bugs:
- make writable roots explicit at container start,
- pass storage roots through env,
- remove `localhost` assumptions from production Terraform,
- stop swallowing session-store path failures.

Phase 2 closes the auth gap:
- add a Next.js route handler proxy at `web/app/api/admin/[...path]/route.ts` that forwards admin requests to the API with `X-Admin-Key` from server env,
- make `AdminWorkspace.tsx`, `SessionInbox.tsx`, `SmartCanvas.tsx`, `TrackBuilderTab.tsx`, `CareerTracksTab.tsx`, `EmployerFactsTab.tsx`, `KnowledgeUpdateTab.tsx`, `BrokenProfilesTab.tsx`, `DocList.tsx`, `BriefGenerator.tsx`, `KnowledgeUpload.tsx`, `TestQueryBox.tsx`, and `ResumeReviewTab.tsx` use that shared transport instead of calling the API directly,
- keep no-key dev mode when `ADMIN_KEY` is unset,
- remove query-param auth from the long-lived path.

This is the right balance because it fixes the real breakage without turning the repo into a full platform rewrite.

## Locked Decisions
- Sessions stay file-backed for this hardening pass, but they move onto explicit persistent storage and a single-writer API task.
- Qdrant stays external to the API task and is addressed by `QDRANT_URL`.
- The steady-state admin path uses header-based auth through a server-side proxy, not query params.
- Query-param access remains only as a dev convenience when `ADMIN_KEY` is unset.

## Success Criteria
- `docker compose up --build` reaches healthy for api, web, and qdrant without manual filesystem fixes.
- The admin UI can load KB and session data in the deployment target where `ADMIN_KEY` is set.
- Session creation works after task replacement or container restart.
- No production config uses `localhost` for a different service.
- Missing or unwritable storage roots fail loudly and leave a clear log line.
- Admin API calls work with `ADMIN_KEY` set without exposing the secret to the browser.
- The API remains stable with one replica and one writer until file-backed state is removed.

## Test Plan
- Local compose: start the stack from scratch, create a session, list sessions, upload a document, and verify restart persistence.
- Production wiring: verify `QDRANT_URL` points at a reachable endpoint, not `localhost`.
- Auth: confirm `/admin` opens without a key only when `ADMIN_KEY` is unset, and confirm protected API calls fail without `X-Admin-Key` in prod.
- Storage: confirm the API logs a clear error when a writable path is missing or permission denied.
- Rollback: if auth migration fails, revert the UI proxy path first while keeping backend guards in place; if storage wiring fails, roll back the task definition and mounts without changing the on-disk paths.

## Dependencies
- Terraform wiring for the real deployment shape.
- Frontend auth helper or server-side proxy for admin API calls.
- Container startup script or init step for writable directories.
- An explicit prod `QDRANT_URL` target, backed by Qdrant Cloud or an equivalent managed Qdrant service.
- Single-writer operational contract for file-backed sessions.
- Terraform resources for `aws_ecs_service.api`, `aws_lb_target_group.api`, `aws_lb_listener.api`, and the security groups that connect ECS to the ALB and EFS mounts.

## Next Steps
1. Add explicit env plumbing for session storage and admin auth, plus a startup init that owns writable directories before dropping privileges.
2. Update the frontend admin fetches to attach auth explicitly and remove the query-param dependency from the steady state.
3. Fix Terraform so the API points at a real Qdrant target and mounts the same writable roots used in local Docker.
4. Change session storage to fail fast on path creation and log the exact root cause instead of returning empty results.
5. Re-run compose, API endpoint checks, and a deployment-target sanity pass.
6. Add the prod AWS resources that are still only implied in `terraform/main.tf`, then verify the ALB, ECS service, and Amplify wiring line up with the env contract.

## What I noticed about how you think
- You did not ask for a random cleanup pass, you asked for deployment hardening after the failures were already visible. That’s the right instinct.
- You kept pushing on environment differences, not just the one Docker bug. That matters, because the same class of issue usually comes back in ECS or preview deploys.
- You cared about file paths and endpoints together. Good. In this repo, those two are welded together whether we like it or not.
