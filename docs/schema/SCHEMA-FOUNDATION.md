Goal: Architecture foundation for extensible, student-queryable career knowledge base

---

## Problem Statement

Current knowledge base (YAML files for employers, career tracks, alumni) stores **critical information as prose narratives** buried in unstructured notes. This creates three problems:

1. **Students can't query it** — "Show me roles that start in June" or "Find alumni from 2020" require human counselor lookup, not direct search
2. **Counselors can't manage it at scale** — Inconsistent data capture. No way to see what's missing or spot duplicates across 50+ employer files
3. **LLM extraction is brittle** — AI has to parse prose for facts. A timeline buried in paragraph 3 of notes breaks if the wording changes

The underlying issue: **there is no schema.** Each file's structure is ad-hoc. Timelines live in `recruiting_timeline` prose. Alumni names hide in `notes`. Interview stages are lists within narrative. Building new features means writing new parsers each time.

---

## What Makes This Architecture Right

You want a **foundation** — a pattern that lets timelines, alumni networks, interview processes, progression paths, and future data types all follow the same structure without rework. The key insight:

Every data type is a timestamped, sourced fact with metadata, observable via Langfuse traces.

Timeline fact: "Summer analyst applications open March 1" (Trace ID: `0193c...`)
Alumni fact: "Aditya Mehta (LLM NUS '18) joined Stripe Compliance in 2020, available for mentoring"  
Interview fact: "Round 3 is live whiteboard, 60 minutes, focus on problem-solving"  

Same pattern. Different content. One schema for all of it.

---

## Constraints

- **Backward compatible** — existing YAML files remain valid; new structure co-exists
- **Human-readable narrative preserved** — counselor context and stories stay; structure supplements, doesn't replace
- **Counselor-friendly** — no code changes required to add facts; structured input form in admin UI
- **LLM-extractable** — prompt can reliably pull facts from prose and write into structured section
- **Incremental adoption** — existing files don't migrate all at once; each gets the new section when next edited
- **Version tracking** — each fact is timestamped with source (counselor email, inferred from LLM, student report) and confidence level

---

## Premises

1. **Prose is not the enemy** — It's where context lives. Structured data supplements it, not replaces it.
2. **Every fact needs provenance** — "Who said this? When? How confident are we?" matters more than thinking everything is equally true.
3. **LLM processing is part of the workflow** — Counselors will use AI to extract facts from interview notes and add them to structured sections. The schema must support that.
4. **The pattern is more important than the fields** — Design the architecture so new field types add cleanly without reworking the whole schema.
5. **Filtering and querying matter from day one** — Structure should enable "show me all employers with January intake" or "find alumni from class of 2020" without extra indexing.

## Approaches Considered

### Approach A: Fact-Table Model (All Structured, No Prose)
Convert everything to normalized facts. Single `facts.yaml` file or flat database schema.

**Pros:**
- Pure structure; LLM can extract reliably
- Queries are trivial (filter by field)
- Can scale to thousands of facts without complexity

**Cons:**
- Loses narrative context entirely
- Counselors lose ability to write stories and nuance
- Breaks backward compatibility completely
- Requires migration of all existing prose
- Feels alienating to humans reading it

**Not recommended.** Violates your core requirement: keep the prose.

### Approach B: Hybrid (Narrative + Structured Sections)
Keep prose narratives at the top level. Add a `structured:` YAML section with timestamped, sourced facts. Counselors edit both; LLM can read and extract.

```yaml
# Example: employer file

employer_name: Stripe Singapore
# ... existing prose fields ...
notes: |
  Fintech unicorn valued at $95B (2023 Series H). Stripe Singapore leads APAC compliance...
  
structured:
  facts:
    - slug: "stripe-intake-summer"
      type: "intake_season"
      value: "June"
      timestamp: "2026-04-12"
      source: "counselor:patricia.lim@smu.edu.sg"
      confidence: 95
    
    - slug: "stripe-timeline-application-window"
      type: "timeline_phase"
      phase_name: "Application Window"
      start_date: "2026-03-01"
      end_date: "2026-05-31"
      timestamp: "2026-04-12"
      source: "inferred_from_job_posting"
      confidence: 80
    
    - slug: "stripe-alumni-aditya"
      type: "alumni"
      name: "Aditya Mehta"
      degree: "LLM"
      school: "NUS"
      graduation_year: 2018
      current_company: "Stripe Singapore"
      current_title: "Head of Compliance Program APAC"
      joined_year: 2020
      available_for_mentoring: true
      contact_method: "via_counselor"
      timestamp: "2026-04-12"
      source: "direct_from_alumni"
      confidence: 100
    
    - slug: "stripe-interview-round-1"
      type: "interview_stage"
      order: 1
      name: "HR Screening"
      format: "phone_call"
      duration_minutes: 20-30
      focus: ["motivation", "background_fit", "logistics"]
      timestamp: "2026-04-12"
      source: "alumni_feedback_aditya_mehta"
      confidence: 85
```

**Pros:**
- Narrative prose stays intact and readable
- Structured facts are timestamped and sourced
- Backward compatible (old YAML works; new section is additive)
- Same pattern works for timelines, alumni, interviews, progression paths
- LLM can reliably parse and write both sections
- Counselors can see what they captured (facts) vs. context (prose)
- Filtering/queries work: filter on structured facts, not on prose
- Easy to add new fact types without schema redesign (just new `type` values)

**Cons:**
- Slight redundancy (same info in prose AND structured)
- Requires counselors to maintain both (though LLM can help with extraction)
- Schema is flexible enough that consistency is on counselors

**Recommended.** This aligns with your goal: foundation architecture + prose preservation + extensibility.

### Approach C: Prose Annotations (Structured Inline Comments)
Keep everything in prose, but add structured metadata as YAML comments or inline tags.

```yaml
notes: |
  Fintech unicorn valued at $95B (2023 Series H). {timeline:intake=June,confidence=95}
  Stripe Singapore leads APAC compliance across 18 countries. {fact:alumni=Aditya Mehta}
```

**Pros:**
- Everything stays in prose; no duplication
- Minimal changes to existing structure

**Cons:**
- Metadata hidden in text; hard to query
- Comments in prose feel awkward
- LLM extraction becomes unreliable (parsing comments is fragile)
- Can't filter on facts without parsing prose
- Doesn't scale to rich fact types (interview stages are multi-field)

**Not recommended.** Defeats the purpose of structure.

## Recommended Approach: Hybrid Structured Schema

**Design the schema as follows:**

### 1. Core Fact Types (Extensible)

Every fact has these mandatory fields:
```yaml
structured:
  facts:
    - slug: "unique-identifier"         # human-readable key
      type: "timeline_phase|alumni|interview_stage|progression_role|skill_requirement|..."
      timestamp: "ISO-8601"              # when captured
      source: "counselor:email|inferred|student_report|document_url"  # provenance
      trace_id: "langfuse-trace-id"      # observability link
      confidence: 1-100                  # how sure are we?
```

Then type-specific fields. Examples:

**Timeline Phase**
```yaml
- slug: "mckinsey-app-open"
  type: "timeline_phase"
  phase_name: "Application Window"
  value: "March 1"                      # or date range
  role_type: "summer_internship"
  duration_days: null
  timestamp: "2026-04-12"
  source: "counselor:dr.patricia.lim"
  confidence: 95
```

**Alumni**
```yaml
- slug: "alumni-aditya-mehta"
  type: "alumni"
  name: "Aditya Mehta"
  degree: "LLM"
  school: "NUS"
  graduation_year: 2018
  current_company: "Stripe Singapore"
  current_title: "Head of Compliance Program APAC"
  career_path:
    - role: "Compliance Officer"
      company: "Stripe Singapore"
      start_year: 2020
      current: true
  available_for_mentoring: true
  contact_method: "via_counselor|direct_email|linkedin"
  topics_willing_to_discuss: ["compliance", "fintech", "career_pivot"]
  last_contact_date: "2026-04-12"
  timestamp: "2026-04-12"
  source: "direct_from_alumni"
  confidence: 100
```

**Interview Stage**
```yaml
- slug: "jane-street-round-1"
  type: "interview_stage"
  order: 1
  name: "HR Screening"
  format: "phone_call|in_person|video|whiteboard|take_home"
  duration_minutes: 20-30
  focus: ["motivation", "background_fit", "logistics"]
  prep_resources:
    - title: "Victor Cheng framework"
      type: "article|video|practice_partner"
      url: null
      cost: 0
  typical_advance_rate: 0.70         # % of candidates who pass this round
  timestamp: "2026-04-12"
  source: "alumni_feedback"
  confidence: 85
```

### 2. Backward Compatibility

Old fields stay. New `structured:` section is additive:

```yaml
# OLD FIELDS — UNCHANGED
employer_name: Stripe Singapore
slug: stripe
tracks: [fintech_compliance]
notes: "Fintech unicorn valued at $95B..."
last_updated: "2026-04-12"

# NEW SECTION — ADDITIVE
structured:
  facts:
    - type: timeline_phase
      ...
```

Files without the `structured:` section work as they always have. Gradual migration as counselors edit each file.

### 3. Fact Queries (for Students & Counselors)

The `structured:` facts enable filtering:

**Student queries:**
- "Show me employers with June intake" → filter facts on type=timeline_phase, value contains "June"
- "Find alumni from NUS 2018" → filter facts on type=alumni, school="NUS", graduation_year=2018
- "What roles have live interviews?" → filter facts on type=interview_stage, format="whiteboard"

**Counselor queries:**
- "Which employers are missing interview stage facts?" → find by `type` absence
- "Which alumni haven't been contacted in 90 days?" → filter on `last_contact_date` < cutoff
- "Low-confidence facts" → filter on confidence < 70 (flag for follow-up)

### 4. LLM Extraction Workflow

Counselor uploads interview notes or employer document → LLM extracts facts:

**Prompt guidance:**
```
From this text, extract facts in structured format:

Facts to extract:
- Timeline phases (when do applications open/close?)
- Interview stages (what's the process?)
- Alumni mentioned (names, roles, years)
- Skills required (what do they want?)

For each fact:
- Use lowercase slug naming: "company-phase-open-date"
- Set confidence: 100 if directly stated, 80 if inferred, 50 if guessed
- Mark source as "document_name" or "inferred_from_context"

Output YAML only. No prose.
```

Counselor reviews extracted facts, can add/edit before committing.

### 5. Schema Evolution

New fact types don't require code changes. Add a new `type:` value, define its fields in admin UI or docs:

```yaml
- slug: "dbs-salary-range"
  type: "compensation"          # NEW TYPE
  role_level: "analyst"
  currency: "SGD"
  base_salary_min: 85000
  base_salary_max: 95000
  bonus_range: "10-25%"
  equity: false
  timestamp: "2026-04-12"
  source: "counselor_survey"
  confidence: 75
```

No schema migration. Type-specific fields are loose (YAML accepts any key). Queries filter on known fields.

## Open Questions

1. **Storage:** Should `structured:` live in the YAML files themselves, or in a separate `facts.json` database? 
   - YAML (in-file): Git-trackable, easy merge resolution, counselor-editable
   - Database (separate): Queryable, easier deduplication, doesn't mix with prose

2. **Deduplication:** How do we prevent the same fact from being captured twice (e.g., timeline "June start" from two different sources)?
   - Approach: fact `slug` is unique; second entry with same slug updates (with timestamp of update)
   - Alternative: allow duplicates with different sources; merge at query time

3. **Versioning:** Do we version the entire file, or track individual fact changes?
   - Track per-fact: more granular, but requires database
   - Track per-file: simpler, git-native, but loses fine-grained audit trail

4. **Confidence semantics:** Is confidence "how sure are we this is true?" or "how complete is this fact?"
   - Current design assumes former (1-100 = certainty)
   - Might need separate "completeness" field (e.g., interview process is 80% complete)

## Success Criteria

1. ✓ **One schema pattern for multiple data types** — Timeline, alumni, interview, progression all use same structure
2. ✓ **Student-queryable** — facts can be filtered/searched without counselor mediation
3. ✓ **LLM-extractable** — prompts can pull facts from prose with 85%+ accuracy
4. ✓ **Backward compatible** — existing files work unchanged
5. ✓ **Extensible** — new fact types don't require code rework
6. ✓ **Sourced & timestamped** — every fact has provenance and confidence
7. ✓ **Human-readable** — prose narratives remain intact and primary
